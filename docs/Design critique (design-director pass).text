Overdrive — propose before build
Per the overdrive skill: no code until you choose. Here are three directions that fit .impeccable (pixel × hyper-modern × activist energy), with trade-offs.

Direction A — Scroll cinema (CSS-first)
Feel: Main sheet doesn’t just slide up—it ** choreographs**: section borders that look like scanlines / pixel steps, progress-linked opacity on the hero, tab panels that enter with @starting-style or staged opacity/transform. Optional scroll-driven parallax on a decorative pixel grid (animation-timeline: scroll() where supported).

Pros: No heavy JS; good performance; aligns “hyper-modern” with motion tied to user scroll.
Cons: Scroll-driven CSS needs Firefox fallback (static layout still strong).
Cost: Medium complexity in CSS, lots of visual tuning in browser.
Direction B — Flag detail as morph (View Transitions API)
Feel: Tap a flag card → the same flag graphic travels and grows into the detail surface (FLIP-style). Feels impossible for a website, very on-brand for “non-conformity.”

Pros: Huge wow-per-pixel; matches functional UI “wow” from the skill.
Cons: Same-document only for broad support; careful with accessibility (reduced motion = crossfade only); polish takes iterations.
Cost: Higher JS + CSS naming discipline (view-transition-name).
Direction C — Live pixel layer (Canvas 2D / low-res pass)
Feel: A chunky pixel buffer (or CSS image-rendering: pixelated + stepped sampling) drives edges of sections, dividers, or a live “CRT window” around the flag strip—not the whole UI. Hyper-modern typography and layout sit around it (juxtaposition).

Pros: Strongest literal 8-bit × modern metaphor; unique.
Cons: GPU/CPU budget and mid-range phones; must lazy-init and pause off-screen; reduced-motion = static bitmap or off.
Cost: Highest engineering + tuning; browser automation essential.
